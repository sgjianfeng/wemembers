# 决策：实体券（印刷代金券 / 抽奖券）

- 日期：2026-07-18  
- 迭代：I1 候选（I0 不插队实现）  
- 状态：**大股东已拍板（含补充）**  
- 试点：Meow BBQ 等

## 背景

门店需要**批量印刷**实体券发给顾客；顾客到店核销。线上券强绑 `customerId`，实体码须「先有码、后可挂人」。

## 大股东答复（汇总）

| # | 决定 |
|---|------|
| 票种 | **到店即用代金券** + **抽奖券**（两套票面，同一套码机制） |
| 无 App | **能**核销 / 能走基础流程 |
| 次数 | **一次用完**（实体形态无多次余额） |
| 范围 | **仅本店**，不可跨店核销 |
| 账号 | **所有实体券均可绑定**；用户扫码 → 注册新账号或登录已有账号 → 绑定后按线上券处理 |
| 抽奖动机 | 绑账号后可随时看大奖信息、**参与抽大奖**（线上 deferred 池） |

---

## 核心原则

1. **码本位**：每张纸 = 唯一码；未绑定前是「持票凭证」，绑定后挂到用户。  
2. **统一可绑定**：代金券、抽奖券 **同一套 claim 入口**（扫实体 QR 即可，不区分「只有抽奖才能绑」）。  
3. **一码一店**：批次/票绑定 **`storeId`**；核销门店必须 = 发码门店。  
4. **一次用完**：无「实体储值多次扣」。多次花余额只走 **线上购券 Voucher**。  
5. **双路径**（绑定是能力，不是门槛）：  
   - **未绑定**：到店店员仍可本店核销（代金）/ 引导绑定（抽奖优先）。  
   - **已绑定**：按 **线上券** 处理（钱包、本店核销、抽奖进池）。  
6. **扫码页唯一入口**：`/c/{code}` → 未登录则注册或登录 → 绑定成功。  

### 为什么「所有实体券都可绑定」更好

| 好处 | 说明 |
|------|------|
| 模型简单 | 一种码、一种 claim，店员/产品不用记例外 |
| 用户路径统一 | 扫纸 → 登录/注册 → 进钱包，话术固定 |
| 数据可跟 | 绑后才有复购、会员、大奖通知；能绑尽绑 |
| 仍兼容无 App | 到店未绑核销代金仍然可以，绑定是升级不是封锁 |

## 本店 only（不跨店）

| 规则 | 说明 |
|------|------|
| 批次 | `PhysicalBatch.storeId` 必填 |
| 核销 | 仅 `session.storeId` / 所选店 = 票的 `storeId` |
| 跨店扫码 | 明确拒绝：「本券仅限 XX 店使用」 |
| 结算 | 本店一次性优惠/开奖，不进跨店结算 |

## 两种票

| 票种 | 未绑定（无/弱账号） | 绑定后（按线上券） |
|------|---------------------|---------------------|
| **代金券** | 本店店员扫码 → **一次核销** → 作废 | 进入用户「可用券」；本店核销一次（同权益） |
| **抽奖券** | 可引导当场绑定；或仅登记手机做 **当场小奖/兑奖**（不进延时大奖池） | **按线上抽奖券处理**：可见奖池/倒计时、参与 **抽大奖**、中奖记在账号上 |

> 绑定 = 扫纸上 QR（或店员帮扫）→ 登录/注册 → `claim` 成功。  
> **绑定后不再当「匿名纸码」处理**，避免双花：码状态 `claimed`，权益落到 `customerId`。

---

## 绑定（所有实体券 · 随时 / 当场）

**统一规则：任意未作废、未核销的实体码，用户扫 QR 都可绑定。**

```
扫 /c/{code}
  → 未登录：选「注册」或「已有账号登录」
  → 已登录：一键绑定到当前用户
  → status = claimed，生成/挂接线上权益
```

```
实体码 status:
  printed → issued → claimed（已绑用户）→ redeemed / void
                 ↘ redeemed（未绑被店员直接核销，不可再绑）
```

| 时机 | 谁操作 | 结果 |
|------|--------|------|
| **随时** | 顾客扫纸上 QR → 注册或登录 → 绑定 | 代金/抽奖都进账号；抽奖可看大奖、进大奖池 |
| **当场** | 店员引导顾客扫同一 QR | 同上，随后按线上券本店核销或开奖 |
| **不绑就用（代金）** | 店员扫码核销 | `redeemed`，本店一次优惠，**不能再绑** |

**防双花：**

- 已 `redeemed` → 不可 claim  
- 已 `claimed` → 店员再扫 = **该用户线上券**，不再匿名核销  
- 一码仅能 claim **一次**、仅绑 **一个** `customerId`  
- claim 与 redeem 互斥加锁

## 抽奖：为什么鼓励注册 + 怎么处理

### 产品叙事（门店话术）

> 「这张抽奖券扫码注册后，可以进 App 看大奖进度，还有机会抽大奖；也可以现在扫一下绑到手机。」

### 分流

| 路径 | 能否进 **延时大奖池** | 通知 |
|------|----------------------|------|
| **未绑定** | **否**（或仅本店即时小奖表，配置可选） | 屏幕 + 口头；中奖登记手机 + 可选 SMS |
| **已绑定** | **是** — 与线上抽奖券同等（权重/规则按批次快照） | App 内进度/中奖记录 + 可选 SMS |

### 「随时查看大奖信息」

- 仅 **已绑定** 用户：活动页 / 钱包显示所属 campaign 的 pool 倒计时、奖品列表。  
- 未绑定：纸上可印「扫码绑定查看大奖」+ 店内海报，**不保证**匿名查池。

### 当场绑定后的处理顺序（推荐）

```
扫码绑定成功
  → 系统创建/关联线上权益（抽奖资格 or 代金权益）
  → 若顾客还在店：可立刻走线上核销/即时抽
  → 大奖：进入 deferred 池，之后在 App 看进度与结果
```

## 与「余额」的边界（不变）

| 场景 | 用什么 |
|------|--------|
| 印刷减 S$10 一次 | 实体代金券（本店、一次） |
| 印刷抽奖资格 | 实体抽奖券 → **鼓励绑定** → 线上大奖 |
| 付费可多次花的余额 | **仅线上购券**，不印刷成实体储值票 |

## 数据概念（实现时）

```
PhysicalBatch
  storeId          // 必填，本店 only
  type             // voucher | draw
  campaignId?      // 抽奖：关联线上活动（大奖池）
  rulesSnapshot    // 面值/即时奖表/是否允许未绑当场开小奖
  quantity

PhysicalTicket
  code             // 唯一，QR 指向 /c/{code}
  batchId
  storeId          // 冗余便于校验
  status           // printed | issued | claimed | redeemed | void
  customerId?      // claim 后填写
  claimedAt?
  onlineVoucherId? / customerCouponId?  // 绑定后指向线上对象
  redeemedAt?
  staffUserId?
  contactPhone?    // 未绑中奖登记
```

核销校验：`ticket.storeId === 操作门店`。

## 印刷内容（每张）

- 店名 + **仅限本店**  
- 票种（代金 / 抽奖）  
- 短码 + QR（claim / 核销同一码）  
- 抽奖票加一句：**扫码注册查看大奖进度**  
- 有效期  

---

## 店员台

1. 扫码  
2. 若 **claimed**：展示用户权益 → 按线上核销/状态  
3. 若 **未 claim**：  
   - 代金 → 本店一次核销  
   - 抽奖 → **优先引导绑定**；若顾客拒绝 → 仅即时小奖路径（若批次允许）或仅收券不做大奖  
4. 跨店 → 拒绝  

---

## 明确不做

- 实体券 **跨店** 核销  
- 实体形态 **多次扣余额**  
- 未绑定实体码 **直接进入** 延时大奖池（无法保证身份与通知）  
- 与跨店结算混用实体一次核销  

线上购券抽奖 V2 **继续独立**；实体抽奖 **绑定后接入** 同一活动池（若 `campaignId` 关联）。
