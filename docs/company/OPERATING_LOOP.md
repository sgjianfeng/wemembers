# 三角色运作环（强制执行）

> 目标：不是「写了角色文档」，而是 **检查 → 发现 → 修复 → 验收 → 再发布**，直到**已发布功能可用**。

## 1. 角色与一票否决

| 角色 | 唯一 KPI | 可一票否决 |
|------|----------|------------|
| **产品** | 验收表勾选率；未绿能力不得对外宣称 | 发布：路径未绿禁止标「可用」 |
| **工程** | P0 缺陷清零；回归自动化绿 | 发布：health / 主路径 E2E 红禁止上线 |
| **市场** | 只用已绿路径演示/试点 | 演示：禁止演示未绿能力 |

大股东只看：**本迭代验收表是否全绿**，不看 PR 数量。

## 2. 每周节奏（建议 2～3 次短循环）

```
① 工程：prod-audit + 主路径 E2E
       ↓ 产出缺陷清单（P0/P1）
② 产品：对照验收表，改状态 / 砍范围 / 定优先级
       ↓
③ 工程：修 P0 → 再跑 audit + E2E
       ↓
④ 产品：手工抽检 P0 路径
       ↓
⑤ 市场：仅用已绿路径做演示 / 验账
```

**未过 ③④ 的功能 = 未发布**（即便代码在 main）。

## 3. 发布门禁（Release Gate）

合并/部署到生产前必须：

| # | 门禁 | 命令 / 证据 |
|---|------|-------------|
| G1 | Health | `curl https://wemembers.store/api/health` → ok |
| G2 | 生产表面审计 | `npm run audit:prod` → pages/apis 无 P0 |
| G3 | 消费者主路径 E2E | `npm run test:e2e:customer-prod` → pass |
| G4 | 产品验收表 | `acceptance-consumer.md` / `acceptance.md` 本迭代项已勾或明确「不做」 |
| G5 | 市场演示脚本 | 只含已绿步骤 |

任一红：**禁止**对用户说「可以用了」。

## 4. 两条产品线（入口分离）

| 线 | 入口 | 主路径 |
|----|------|--------|
| **消费者** | `/` | 注册 → 登录 → 首页 → 购券专链 → 余额 |
| **商家** | `/for-business` → 注册/登录 → `/business` | 门店 → 活动 → 核销 |

混在同一首页的双 Tab **已废除**（决策 D-P1-3）。

## 5. 缺陷分级

| 级 | 定义 | SLA |
|----|------|-----|
| **P0** | 主路径不可用（注册/登录/买券/核销/5xx/错跳角色） | 24h 内修或回滚 |
| **P1** | 可用但严重受损（文案 key、403 白屏、短信偶发） | 本迭代内 |
| **P2** | 体验/增长（找店、关注、最近） | backlog 排序 |

## 6. 本迭代「已发布即必须可用」清单

见：

- `docs/roles/product/acceptance.md`（I0 商家主路径）


结果贴进 `defect-backlog.md` 顶部「最近审计」。
