## 一、首页 Landing Page（/）[7.5 分]

### PM 视角

**做得好的：**
- 三支柱卡片（券·会员·抽奖）信息层次清晰，icon + 标题 + 功能点列表，扫一眼就知道产品做什么
- 统计数据区"3大模块·5分钟上线·S$0月费"直击商家招聘痛点
- CTA 按钮"免费开始使用"明确，二次引导"了解更多↓"给观望用户台阶
- 三步流程图（注册→创建→营业）简洁易懂
- 中英双语切换自然，没有明显翻译生硬感

**需要改进的：**
- Hero 区太像纯 SaaS 官网，**完全面向商家**。文案："一站式商户营销平台"。但实际场景中消费者占比更高（扫码领券、朋友分享链接）
- 缺少产品真实截图/操作预览。三根大柱子全是文字描述，缺乏说服力
- 底部 CTA 和顶部 CTA 完全重复
- 没有社交证明（商家数量、券发放量、用户评价）

**优化方案：**
1. Hero 区增加双角色切换 Tabs：「我是商家」/「我是消费者」
2. 每根柱子加一张 mockup 图或 GIF 动图
3. 底部 CTA 替换为数字信任证明（"已服务 X 家商户 · 已发放 Y 张券"）
4. 消费者视角隐藏三步流程，替换为"附近热门优惠"卡片

### 客户视角

> 「页面好看，但我第一次打开的时候不知道我是来领券的还是开店的。三根柱子下面的功能描述很详细，但我更想看到现在有什么优惠可以领。附近商家、热门券、搜索框都没有。」

**痛点：**
- 消费者入口缺失。我如果是被朋友分享过来的，看到首页会懵
- 颜色不错（紫蓝渐变），但有质感但缺少功能引导

### 架构师视角

**代码问题：**
- `"use client"` 但没有充分利用客户端能力——所有内容都是硬编码在 TS 文件里的 i18n 对象（~150行），维护成本高
- `fetch("/api/auth/me")` 在 `useEffect` 里调用，导致已登录用户会先看到"免费注册"按钮然后闪烁变回 Dashboard。应该用 cookie 在服务端判断登录态
- i18n 内容混在一个大对象里（`content.zh` / `content.en`），应该抽成独立的 JSON 文件
- 代码中通过 `lang === "zh" ? ... : ...` 内联了一些中文文案，没有走 i18n，不一致

## 二、登录/注册流程（/auth/login, /auth/register）[8.2 分]

### 得分最高的模块，但仍有提升空间

**做得好的：**
- 手机号 + 验证码，两步完成，无需记住密码
- `CodeInput` 组件（6位分格输入）交互体验好，每个数字独立显示
- 注册页有"验证码已发送到 XXX"的确认提示
- 登录成功自动按角色路由（admin→/admin, business→/business, customer→/home）

**需要改进的：**
- 没有第三方登录（Google / Apple / 微信）。新加坡市场 Apple 和 Google 是基本需求
- 登录和注册页长得几乎一样（Logo + 表单），视觉上没有区分。注册页应该增加 benefits checklist
- 注册时用户选角色，但这个引导非常弱——用户可能是直接走「快速注册」进来的，不知道自己选了什么
- "发送验证码"按钮没有倒计时，用户可以重复点击

**优化方案：**
1. 登录/注册页视觉差异化：注册页增加"S$0 免费注册，注册即赠 100 Token"等福利卡片
2. 发送验证码后增加 60 秒倒计时
3. 添加 Google / Apple 第三方登录按钮
4. 注册流程分两步：第一步填手机号 → 第二步选角色（商家 or 消费者），每一步有独立视觉引导

### 架构师视角

- 客户端组件，职责单一合理
- `CodeInput` 组件封装良好
- 跳转 URL 硬编码（`/home`, `/business`, `/admin`），应该统一到路由常量

## 三、用户首页 Home（/home）[7.0 分]

### 信息密度过高，缺少个性化

**做得好的：**
- 会员身份卡设计亮眼，蓝色渐变 + 积分大字 + 等级徽章 + 进度条
- 热门优惠卡有智能打分函数（`couponAttractiveness`），考虑折扣力度、稀缺性、流行度
- 持券数、省钱总额、签到天数三个数据直观展示
- 每日签到模块促进 DAU
- 倒计时卡片让抽奖活动有紧迫感
- 省钱自动统计 + 鼓励文案（"本月已省 S$XX！太棒了，继续保持！"）

**需要改进的：**
- **信息过载**：身份卡 → 代币条 → 倒计时卡 → 热门券 → 省钱卡 → 签到 → 发现更多，用户不知道先看哪个
- **没有个性化推荐**：所有人都看到同样的热门券，没有「离我最近」「你常去的那家」「因为你领过 X 所以推荐 Y」
- 积分进度条做得好（"距离银卡会员还差 XXX 分"），但没有引导用户怎么做任务赚积分
- 热门优惠只展示前 3 张，点击"发现更多"需要滚动很远才看到
- 倒计时卡显示抽奖倒计时，但点进去发现要上传收据，不是免费的——有预期落差

**优化方案：**
1. 首页信息分层：Level 1（身份卡 + 核心操作）→ Level 2（个性化推荐）→ Level 3（签到 + 省钱统计）
2. 「离我最近的优惠」替代纯热门排序
3. 积分进度条下面增加一个小任务列表（"再签到 2 天可获额外 20 分"）

### 架构师视角

**代码亮点：**
- Server Component + `Promise.all` 并行 6 个查询，性能好
- `couponAttractiveness` 打分函数考虑了多维度

**代码问题：**
- `entries` 查询在第一次查询结果回来后单独再查（两次数据库往返），应该合并到 `Promise.all` 中
- `couponAttractiveness` 打分逻辑纯静态（没考虑用户历史行为），返回排序结果偏随机
- `featuredDeals` 取前 3 个，`VoucherTabs` 从第 4 个开始——UI 上这两块视觉分离度不够，用户容易以为它们是同一组
- 每天都要查 `validUntil: { gt: new Date() }` 会影响缓存效率，应该预计算 `expired` 状态

## 四、发现 & 领券（/shop/[slug], /store/[slug], /coupons/[id]） [6.2 分]

### 券详情页信息不足，是用户流失关键点

**做得好的：**
- 券详情页 Hero Card 视觉醒目——橙红渐变背景 + 大额数字
- 信息字段齐全：类型、面值、所需积分、最低消费、有效期、库存、限领数
- 已领取状态展示清晰（变绿 + "已领取" Tag）
- 支持 Gift 转赠（券可赠送好友）
- Server Component，性能好

**严重不足的：**
- **店铺页和门店页几乎完全重复**——URL 不同但展示内容一样，应该合并
- 券详情页没有「使用说明」「适用门店」「不适用商品」——这是新加坡消费者最关心的
- 积分不够的用户只能干瞪眼。没有"如何快速攒分"的引导
- 券详情页没有「分享给好友」的按钮。虽然有 gift 功能但券详情页上看不到
- 缺少商家品牌视觉（Logo、封面图、营业时间）
- 没有「猜你喜欢」的相似券推荐

**优化方案：**
1. 券详情页增加「适用门店列表」「使用条款」两个折叠区
2. 积分不够的用户显示引导："您还差 XX 积分，去签到或消费可快速获得"
3. 底部增加"你可能也喜欢"的相似券推荐（同商家、同类型、相近积分）
4. 合并 /shop 和 /store 功能，/store 重定向到 /shop
5. 增加商家品牌卡（Logo + 介绍 + 营业时间）

## 五、钱包 & 核销（/wallet, /redeem/[id]）[6.7 分]

### 核销页体验好，但钱包信息密度太低

**做得好的：**
- 三 Tab 切换（可用/已用/已过期）清晰
- 核销页（/redeem）设计干净：大额数字 + QR 码占位 + 4位分格码 + 提示文案
- 支持转赠（Gift），入口在底部 Sheet 中
- 已过期券自动变色提醒

**需要改进的：**
- **钱包中每张券卡片只有数字，没有颜色标识、没有适用门店**
- 已过期券占据视觉空间但无价值。建议加"过期券换积分"或直接隐藏
- 核销页只有占位 QR（📱 emoji），没有真实二维码。应该集成 qrcode 库生成真实二维码
- 核销页没有「在哪个门店使用」的提示
- 没有核销历史列表——用户核销后找不到记录

**优化方案：**
1. 钱包券卡片增加颜色区分（固定金额=蓝、折扣=橙、免单=紫）
2. 增加「适用门店」标签
3. 已过期券移到独立页或默认折叠
4. 核销页集成真实二维码生成（npm qrcode）
5. 钱包增加「核销记录」子 Tab

## 六、抽奖活动页 V1（/draw/[slug]）[7.2 分]

### 视觉冲击力强，但操作摩擦大

**做得好的：**
- 橙红渐变背景 + emoji 丰富，视觉热闹有抽奖氛围
- 双模式切换（即时抽 vs 延迟抽），每个模式都有一句话说明
- 倒计时时钟增加紧迫感
- 奖池双轨展示（即时池 + 大奖池）+ 进度条透明
- 即时中奖结果用 🎉 emoji + 奖品名反馈，延迟抽则展示券号
- 已领取券票展示 myTickets 展开/收起交互

**需要改进的：**
- **收据上传只有金额输入框**——用户需要手动输入消费金额。没有拍照上传、没有 OCR 识别
- 没有清晰展示"为什么选即时抽"vs"为什么选延迟抽"的对比（概率、奖池大小）
- BYD 解锁进度不够显眼——这是最大卖点，应该有独立的大卡片
- `minSpendSgd = (campaign.receiptMinSpend || 5000) / 100` 默认门槛 S$50，页面中只在一行小字里提到

**优化方案：**
1. 收据上传改为拍照 + OCR 自动识别金额
2. 双模式对比表（即时=小额但当场中、延迟=大奖但等开奖）
3. BYD 进度独立大卡片展示，视觉上突出
4. 增加"你已有 X 张券，中奖概率约 X%"的个人统计

### 架构师视角

**代码问题：**
- Client Component + state 管理，逻辑清晰
- `submit()` 在前端用 `Math.round(parseFloat(amount) * 100)` 算 receiptAmount，后端再用 `Math.floor(receiptAmount / minSpend) * ticketsPerUnit` 算 ticketCount —— 存在前后端逻辑不一致的风险。应该让后端返回 ticketCount
- 即时抽奖的结果展示和延迟抽的结果展示用同一个 Card，区分度不够

## 七、代金券抽奖 V2（/voucher/[slug]）[6.0 分] <⚠️ 最高优先级>

### 核心变现页面，但仍然不成熟

**做得好的：**
- 已有架构基础：`PoolDashboard` 组件展示三档大奖进度（iPhone/MacBook/BYD）
- `VoucherTierSelector` 组件提供三个档位选择（S$20/50/100），带 icon 和 badge
- 奖池状态 API（`/api/campaign/pool-status`）返回完整的池数据和倒计时
- 购买后 100% 中奖（即时奖）+ 大奖池入口展示
- 分享加成按钮（share boost）已实现
- `resolveTier` 自动判断券面金额所属档位
- 余额预览（"消费后余额 S$XX"）实时计算

**严重不足的：**
- **用户看不到"买这张券能中什么"** ——即时奖品列表（S$0.50~S$20 七个档位）没有在页面上展示
- **三档券的区别不直观** ——S$20 和 S$50 的即时奖上限不同（S$2 vs S$8），但用户看不到
- **分享加成只显示了按钮，没有引导** ——用户不知道分享能让中奖概率翻倍
- 奖池进度数据来源正确（`/api/campaign/pool-status`），但进度条下面没有解释"这些钱从哪来"
- Small 券买后不参与大奖池的提示太小，用户容易误解

**优化方案：**
1. **增加即时奖品预览区**：选中券面金额后，下方展示"你可能赢得"的奖品列表 + 概率条
2. **三档券对比表**：展示每个档位「即时奖上限 · 大奖池权重 · 推荐场景」
3. **分享引导强化**：中奖结果弹出时如果有大奖池资格，自动引导分享（"分享给好友，中奖概率 +50%！"）
4. 奖池进度条下方增加解释文案："奖池资金来自每张券的 20%"
5. 购买按钮改为两段式："确认购买 S$XX → 支付"
6. 购买成功后自动弹出分享引导（而不是底部小按钮）

### 架构师视角

**代码亮点：**
- V2 系统架构清晰：`draw-v2.ts`（纯算法库）+ `PoolDashboard`（展示组件）+ `VoucherTierSelector`（选择组件）+ `/api/campaign/pool-status`（数据 API）
- `estimatePoolCountdown` 算法好：按日均速度预测倒计时，只加速不减速
- 分享加成已实现前后端

**代码问题：**
- `fetch("/api/draw/${slug}")` 拿的是 V1 campaign 数据，但 V2 用的是 `lucky_draw_v2` 类型。如果 campaign 是 v2 类型，这个 API 返回的字段可能不全
- `PoolDashboard` 组件无条件渲染（即使没有 countdown 数据），应该用 Suspense + loading skeleton
- 目前只能选 S$20/50/100 三个档，但 `resolveTier` 支持连续值。应该有自定义金额输入
- V1 和 V2 两套抽奖系统完全独立，共享代码少——维护成本翻倍

## 八、会员卡（/card/[businessId]）[5.3 分] <⚠️ 最低分>

### 现在是空的列表页，不是会员卡

**严重问题：**

这个页面目前只是一个商家 membership 列表——展示了用户在哪些商家有会员资格（店名 + 积分 + 访问次数）。但这完全不是"会员卡"的体验：

- 会员卡应该像一个真卡——有卡号、等级颜色、权益列表、积分、QR 二维码
- 目前只是一个列表，点进去跳到店铺页，没有任何会员专属体验
- **没有展示会员权益**——金卡会员有什么特权？这在哪都看不到
- 和 Profile 页的等级徽章功能重叠

**优化方案（彻底重做）：**
1. 每张卡做成真卡样式：渐变背景 + 等级图标 + 卡号 + QR 码
2. 每张卡展开后显示权益列表（"本店消费 9 折""生日免费饮品""优先预约"）
3. 显示该商家下的积分进度（距下一级还差多少）
4. 支持 Apple Wallet / Google Pay 添加

## 九、推广系统（/promoter, /promoter/withdraw, /p/[code]）[6.7 分]

### 出乎意料地好——未激活引导做得很棒

**做得好的：**
- **未激活页面设计优秀**：💸 emoji + 4 步说明（生成链接→分享→核销到账→提现）+ 一行一个 icon 卡片，信息传达效率极高
- 激活后「推广中心」展示收益概览（今日 + 可提现 + 累计 + 等级）
- 已生成的推广链接列表显示详情（所属商家、券名、推广码、点击量、核销量、一键复制）
- 可推广的券列表展示奖励类型（现金/实物/抽奖）和预计收益
- 最近收益流水展示清晰
- 提现页面有余额显示

**需要改进的：**
- 激活后的页面顶部收益卡用绿色，和首页蓝色系不统一
- 推广链接列表只展示文字，没有显示链接状态（是否仍可用、是否过期）
- 没有推广素材——用户需要自己写文案。应该提供默认海报和文案模板
- 提现渠道只提到微信/支付宝，但对新加坡市场不支持。应该展示 Stripe 提现
- 提现页面没有历史记录

**优化方案：**
1. 推广中心增加"生成海报"按钮——一键生成带二维码的推广图
2. 提供默认分享文案模板（"这家店的券超值！限量 XX 张，手慢无！👇"）
3. 推广链接列表增加状态标签（✅可用 / ⚠️即将过期 / ❌已失效）
4. 提现渠道改为 Stripe

### 架构师视角

- 推广激活 API `/api/promoter/activate` 同时服务 GET 和 POST，职责清晰
- `generateLink` 函数处理单个券的推广码生成，逻辑好
- 但是 Dashboard 数据每次重新 fetch，没有缓存

## 关键改进路线图

### 🔴 P0 — 本周必须修（影响核心变现和用户体验）
